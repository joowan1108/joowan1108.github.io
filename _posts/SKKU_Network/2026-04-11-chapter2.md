# Application Layer

TCP/IP Protocol의 제일 user와 맞닿은 layer가 Application layer이다.

Application layer는 Internet 사용자에게 service을 제공하는 역할을 한다

Application Layer에서 사용되는 protocol은 두 가지로 나눌 수 있다.

1. Standard Application-Layer protocol
    - Internet authority에서 공식화한 것
    - ex) HTTP, FTP, SMTP, POP3, TELNET

2. Nonstandard Application-Layer protocol
    - Internet authority에서 공식화하지 않은 것
    - Private use only

## Application Layer Paradigm

Application Layer Paradigm이란 network의 구조라고 보면 된다. 각 host는 어떻게 배치될 것이고 각자 어떤 역할을 할 것인지를 정의한 것이라고 보면 된다.

Application Layer paradigm으로는 3가지가 존재한다.

1. Client-Server paradigm

    Client-Server paradigm은 host의 역할을 server와 client으로 나눈 것이다.

    **Server**
    - Always on
    - Client의 요청에 대한 답변을 주는 host
    - IP address가 영원함

    **Client**
    - Server와 연결을 시작하는 host
    - Client들끼리 서로는 직접 통신을 하지 못하고 항상 server을 거친다

    ex) WWW (World Wide Web), FTP, Remote login (TELNET, SSH (Secure
    SHell)), E-mail

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg7.png)



2. Peer-to-Peer paradigm

    모든 host의 역할이 "Peer"으로 고정되어있다.

    - No always-on server
    - 직접 통신 가능
    - Peer끼리 가끔 연결되고 IP address는 변한다
    - Highly distributed and decentralized
    - High scalability, increase service capability
    - 보수하기 어려움

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg9.png)


3. Mixed paradigm

    Client-Server와 Peer to peer 성격을 둘 다 가진 paradigm이다.

    Client은 server로부터 정보를 가져오지만 client끼리도 정보를 주고 받을 수 있는 paradigm이다.

    ex) Nate, Yahoo messenger


## Socket: Application Programming Interface of Application Layer

Application layer는 lower four layers와 정보를 주고 받기 위해 socket이라는 API을 사용한다.

Socket API는 application layer가 operating system과 소통하기 위해 존재한다.

*이때 socket은 application program이 생성하는 data structure이다.*

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg13.png)

**Socket을 Client-Server paradigm 관점에서 설명해보자**

이 socket은 server와 client끼리 직접적인 logical connection이 존재하는 것처럼 느끼도록 해주는 역할을 한다. 즉, client는 socket이 request을 받고 response을 주는 객체라고 생각하고 server는 socket이 request을 하고 response을 받아내는 객체하고 생각하게 된다는 것이다.

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg15.png)

Client와 server는 양방향의 소통을 하기 때문에 한 쌍의 address (local, remote)가 필요하다. 앞서 말했듯이 client-server paradigm은 client와 server의 socket을 통해 소통을 하기 때문에 이 address는 사실 socket address이다.

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg16.png)


**Server 입장**

- Server (Local) Socket Address

    $\rightarrow$ Operating System이 정해준다

    $\rightarrow$ Server가 실행되는 computer의 IP Address을 갖는다

    $\rightarrow$ Port는 server가 실행될 때 정해진다

    $\rightarrow$ Server가 실행되는 동안에는 고정된다. 


- Client (Remote) Socket Address

    $\rightarrow$ Client가 server에 접근하려고 할 때 정해진다

    $\rightarrow$ Client마다 고유하다


**Client 입장**

- Client (Local) Socket Address

    $\rightarrow$ Operating System이 정해준다

    $\rightarrow$ Client가 실행되는 computer의 IP Address을 갖는다

    $\rightarrow$ Port는 client가 server에 접근하려고 할 때 정해진다

- Server (Remote) Socket Address

    $\rightarrow$ 사용자가 IP address와 port number를 직접 정해줘야 한다. (ex) url의 domain name을 통해)


## Using services of Transport layer

Application layer는 socket을 통해 나머지 layer와 연결한다고 하였다. Application layer에는 물리적인 connection이 없기 때문에 하위 layer의 service을 통해 물리적인 연결을 얻어야 한다. 다만 이 과정이 socket으로 인해 application layer 자체는 알지 못하는 것이다. 

그렇다면 socket은 하위 layer들과 어떻게 연결되는 것일까? 

$\rightarrow$ Socket이 사용할 언어 (transport layer의 service)을 선택함으로써 socket이 transport layer와 연결할 수 있게 되는 것이다.

이 service를 transport layer protocol이라고 하고 종류는 크게 3가지이다.

1. UDP (User Datagram Protocol)
    - Connectionless, unreliable, datagram
    - Message의 단위는 **datagram**으로 encapsulated된 packet이다.
    - Connectionless: Message을 주고 받는 양끝 객체들끼리는 logical connection이 없다
    - datagram은 독립적이다.
    - small data usage, speed > reliability, multimedia에 적합

2. TCP (Transmission Control Protocol)
    - Connection oriented, reliable, byte stream
    - Connection oriented: Data을 주고 받는 양끝 객체들끼리 logical connection이 이루어진다. 
        $\rightarrow$ 이 connection은 handshake process을 통해 이루어진다
    - Long messages, reliability에 적합 (File transmission)



3. SCTP (Stream Control Transmission Protocol)
    - Connection oriented, reliable, message oriented
    - 여러 multiple network connection을 통해 mutli-stream service가 가능하다

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg23.png)


## Standard Client-Server Application Protocol

Client-Server paradigm을 적용한 application들과 각 application을 위한 protocol을 보자

### World Wide Web protocol: HTTP

**World Wide Web이란**

World Wide Web은 distributed client-server service으로 browser(client)가 web server (server)로부터 web page을 얻게 해주는 service이다.

- Browser는 server로부터 갖고 온 web page을 조회하는 client이다. Browser는 controller, client protocol, 그리고 interpreter로 구성되어 있다.

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg29.png)

- Web server는 client가 요청한 web page을 반환하는 역할을 한다. 효율을 높이기 위해서 요청된 web page들을 cache에 보관할 때도 있다.

**Web Page**
Web page는 identifier으로 url이 사용되며 protocol, host(IP address of server / domain name), port (16 bit, predefined), path(Operating system에서의 위치)으로 구성된다. 

> protocol://host/path
> protocol://host:port/path

Web page는 세 종류로 나뉜다.

1. Static
     - fixed content
     - server에 저장되어 있고 copy만 가져가는 것임
     - Language: Hypertext Markup Language (HTML), Extensible Markup Language
    (XML), Extensible Style Language (XSL), and Extensible Hypertext Markup
    Language (XHTML)

2. Dynamic
    - 요청 받을 때마다 web server에서 생성
    - Language: Common Gateway Interface (CGI), Java Server Pages (JSP), Active Server Pages (ASP), and ColdFusion

3. Active
    - Client에서 실행되고 있는 프로그램
    - Language: JavaScript or Java (Applet)

**World Wide Web protocol: HTTP**

World Wide Web application을 사용하기 위해서는 HTTP protocol을 사용해야 한다. HTTP protocol을 통해 web server에서 web page들을 가져올 수 있는 것이다.

Server는 port number 80을 고정으로 사용하고 client는 임시 port number을 사용한다.


#### HTTP 특징

HTTP는 TCP 기반이고 HTTP에서 client와 server가 연결되는 방법에는 두 가지가 존재한다.

**Connection Type**

1. Nonpersistent Connections
    각 request/response마다 하나의 TCP connection을 수행하는 방법이다. 하나의 request을 할 때마다 한번의 connection을 해야 하기 때문에 한 request 하기 전에 3번의 handshake을 해야 하고 response을 받으면 close을 해야 한다. Close하기 위해서는 또 3번의 handshake을 해야 한다.

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg35.png)

2. Persistent Connections
    Persisten connection에서는 server가 response을 보내더라도 계속 connection을 유지한다. 

    Connection close는 timeout이나 client의 request으로 가능하다.

    각 response에 server는 data의 길이를 포함시킨다. 하지만 web page가 dynamic하거나 active일 때는 data의 길이가 정해져있지 않기 때문에 data의 길이에 대한 정보를 client에게 알려주기 위해 data을 다 보낸 뒤에 server을 닫는다.

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg37.png)
 

HTTP을 사용한 예시로 GET method와 PUT method가 있다.

1. GET: Server에서 web page을 가져오기
    HTTP에서 request는 method 이름, URL, 그리고 사용한 http version이 포함된다.
    Request body는 비어있다.

    GET의 response는 response의 status 정보 + 4 lines of header (date, server, content encoding, length)가 포함된다.

2. PUT: Server에 web page을 post하기
    HTTP에서 request는 method 이름, URL, 그리고 사용한 http version, 4 lines of header가 포함된다. 또, request body에는 web page의 내용이 들어있다.

    Response message에는 response의 status 정보 + 4 lines of header가 포함된다. 생성된 document는 response body에 포함된다.

**Cookies**

World Wide Web의 Web server는 cookies을 통해 이전에 방문한 client들의 정보를 기억할 수 있다.

Cookies는 server가 client에게 response을 보낼 때 cookies을 생성하고 response에 포함된다. 

Client에 도착한 client는 client의 cookie directory에 저장된다. 이 cookie는 client가 server에 다시 request을 전송할 때마다, cookie directory에서 호출되어 request에 포함된다.

이 과정을 통해 server가 client을 기억할 수 있게 된다.

*이때, cookie의 내용은 browser나 user에게 공개되지 않고 server만 생성하고 처리할 수 있다.*

**Web Caching: Proxy Server**

HTTP는 proxy server을 통해 최근 request들에 대한 response을 저장할 수 있다. 이 proxy server는 cache처럼 작동하여 request가 올 때 proxy server에 그 request에 대한 response을 한 적이 있는지 확인한 다음, 있다면 proxy server에서 직접 response을 전송하고 없다면 web server에서 찾는다. 

$\rightarrow$ 이를 통해 web server에 부담을 줄이고, traffic 감소, improve latency (반응 속도)

Proxy server는 보통 client 측에서 저장된다. 따라서, 한 LAN 컴퓨터에 proxy server을 실행하면, LAN 안팎을 오가는 data 수를 줄일 수 있다.

이 proxy server 내용은 어떻게 update할까?
- 일정 시간 이상 유지되는 내용은 계속 저장
- 마지막으로 modified된 날짜를 header에 저장하여 이를 기준으로 삭제/저장

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg52.png)


##### HTTP 진화

HTTP은 본래 static web page을 위해 디자인 된 것이지만 image/script 등 resource가 많이 필요한 web page들이 많아졌다. 

따라서 bandwidth와 processing load을 늘려야 했다.

또, HTTP은 원래 sequential protocol으로 동시에 fetch을 하지 못했다.


**HTTP/2**

- Binary Protocol

    기존의 HTTP는 text을 통해 header을 구성하였다. 이로 인해 parser는 header의 전체 길이를 바로 알 지 못했으며 header field들은 중복되는 내용이 많았다.

    HTTP/2의 binary protocol은 text로 표현되던 header을 binary로 표현함으로써 request와 response을 처리하는 효율과 속도를 높였다.

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg54.png)

- Stream and multiplexing

    각 request/response 쌍을 stream 값으로 identify하여 client 측에서 여러 response가 오더라도 각 response가 어떤 request을 위한 것인지 바로 알 수 있게 되어 response을 병렬으로 여러 개 받을 수 있게 되었다.

- Stream priority

    Client가 server에게 어떤 내용을 더 우선적으로 고려했으면 좋을 지를 보낼 수 있게 되었다. (ex: web page을 보여줄 때 HTML, CSS, JSP을 image보다 우선시해라)

    HTTP/2는 stream weight와 stream dependency을 통해 priority을 알아낸다

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg56.png)


- Server push

    Client가 나중에 무엇을 요청할 지에 대한 사전 정보를 server가 알 수 있게 되었다.

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg57.png)


하지만 HTTP/2도 단점을 가지고 있다. HTTP는 본래 TCP 기반이기 때문에 TCP에서 packet들을 순서대로 합친 뒤 application layer에 보낸다. 하지만 HTTP/2가 제공하는 stream and multiplexing으로 인해 각 stream이 독립적으로 변한다. 아무리 request/response가 stream 값을 통해 indexing 되어 있어도 이는 application layer만 해석할 수 있는 것이고 아래 layer인 TCP는 이 정보를 알 지 못한다. 즉, TCP에게는 그저 byte stream일 뿐이다. 따라서 application layer는 각 packet이 어떤 순서인지 알아도 TCP 입장에서는 자기가 순서를 맞춰서 전달해줘야 하기 때문에 1번 packet이 loss 되었다면 2,3,..이 와도 discard하고 1번 packet이 재전송 될 때까지 기다리게 된다. 이로 인해 모든 stream이 멈추게 된다.

**HTTP/3**

HTTP/3은 Quick UDP Internet Protocol (QUIC)을 통해 이 문제를 해결한다

### File Transfer Protocol: FTP

한 host에서 다른 host에게 file을 복사하기 위해 필요한 protocol이다. 

HTTP을 통해서도 file을 주고받을 수 있지만 FTP는 다양한 형식의 file과 큰 용량의 file에 더 적합하다.

이때, file transfer란 셋 중 하나다.
1. Retrieving file (server -> client)
2. Storing file (client -> server)
3. Directory listing (server -> client)
 
FTP server는 server control process와 server data process으로 구성된다. Control process끼리는 control connection을 통해, data process끼리는 data connection을 통해 연결된다.

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg59.png)

두 연결 종류는 서로 다른 lifetime을 가지고 있다.

Control connection은 FTP session이 시작될 때 연결되고 session동안 계속 연결되어 있는다

반대로 Data connection은 파일 transfer 할 때마다 연결이 되고 다 보내지면 바로 closed된다.

또, 둘은 생성 과정이 다르다. 

Data connection은 서버가 먼저 port 20에서 open하여 client을 기다린다. 반대로 control connection은 순서가 반대여서 client가 기다리는 역할을 한다.

Control connection의 순서는 다음과 같다.

1. Client가 먼저 임시 port을 통해 passive open을 시작
2. Client가 port number 21으로 server에 임시 port number을 전달
3. Server가 port 20번과 전달받은 port number을 연결하여 active open을 시작하여 실제 데이터 연결이 시작

이때, port 20과 client의 port가 연결되더라도 control connection을 통해 어떤 파일을 주고받을 지 결정된다.

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg65.png)

RETR 명령어를 실행할 때만 data connection이 성립된다. 반대로 control connection은 항상 되어있다.

### Email Protocol

Email은 HTTP나 FTP와 달리 one-way transaction이다. 즉, sender와 receiver가 동시에 연결 안 되어있어도 된다.

즉, sender와 receiver 모두 오직 client program만 실행하고 있어도 중간에 있는 서버들이 client-server paradigm을 통해 메세지를 전달한다는 것이다.

이 내용은 email system의 구조를 통해 알 수 있다.

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg71.png)

**User Agent**

- User Agent (UA)는 user에게 메일 전송 / 받기 과정을 쉽게 해주는 service을 제공한다. 
- UA는 mail의 생성, 읽기, 답장, mailbox 관리 등을 돕는 user computer에서 실행되는 프로그램이다.
- *UA는 command-driven UA (mail, pine)와 GUI-driven UA(Outlook)으로 나뉜다.*

메일을 전송하기 위해서 UA는 envelope와 message(header + body)으로 구성된 형식을 생성한다.

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg73.png)

이때, address는 local part와 mail server의 domain name으로 구성된다.

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg74.png)

*Email은 한 이름으로 다양한 Email 주소들을 표현할 수 있다. 이 기능을 mailing list이라고 한다. 이 mailing list의 모든 사람들에게 mail을 보내기 위해서는 그 사람 수만큼의 message을 만든 뒤, Message Transfer Agent (MTA)에 전송되어야 한다.*

**Message Transfer Agent**

Email system은 세 client-server application (두 번의 MTA 간의 application + 한 번의 MAA 간의 application)으로 분석할 수 있다.

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg75.png)

이 application들이 사용하는 protocol도 다르다. MTA client와 MTA server 간에는 SMTP (Single Mail Transfer Protocol)을 사용하고 MAA client와 MAA server 간에는 POP / IMAP을 사용한다.

SMTP는 client $\rightarrow$ server, server $\rightarrow$ server에 사용되고

POP/IMAP은 server $\rightarrow$ client에 사용된다고 보면 된다.

SMTP에서 메일 transfer 과정은 3 단계를 거쳐야 한다.

![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg80.png)

1. Connection Establishment
    SMTP client가 port 25으로 TCP connection을 하여 SMTP server가 connection phase 시작
2. Message Transfer
    Sender와 한 명 또는 여러 receiver 간의 single message 교환
3. Connection Termination
    SMTP client가 connection 끊기

**Message Access Agent**

POP3와 IMAP은 MAA Server로부터 MMA Client으로 email을 가져오기 위해 사용되는 protocol이다.

- Post Office Protocol (POP3)
    - 간단하지만 제한된 기능을 가진다.
    - Client용 POP3 software는 recipient computer에 설치되고 server용 POP3 software는 메일 server에 저장된다.

    ![joowan1108]({{site.url}}/images/SKKU_Network/chp2pg83.png)

- Internet Mail Access Protocol (IMAP)
    - IMAP은 Email을 다운로드 하기 전에 header를 확인하고 내용을 검색할 수 있다.
    - IMAP은 Email을 부분적으로만 다운받을 수 있어 bandwidth가 제한되거나 multimedia가 들어있을 때 유용하다
    - Mail server에서 mailbox을 편집할 수 있다.

    - A cookie is a small piece of text file stored on a user's computer in the form of a name-value pair
    - Cookies are used by websites to keep track of visitors. The most common
    example of using a cookie is to store User information, User preferences, Password Remember Option, etc

5. Provide the full name of the acronyms: SMTP, POP3, IMAP
    - SMTP: Simple Mail Transfer Protocol
    - POP3: Post Office Protocol version 3
    - IMAP: Internet Mail Access Protocol
6. Consider SMTP, POP3 and IMAP. Are these stateless protocols? Why or why not?
    - All three are stateful protocols
    - They maintain store the past information for further uses:
    - SMTP: stores information of the sender, recipients for sending the email within a session
    - POP3 and IMAP: maintain a working session of users
